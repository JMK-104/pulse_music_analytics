# 5. Table Specifications

## 5.1 Purpose

This section provides a high-level catalog of the operational database tables and their business responsibilities.

Detailed column definitions, data types, constraints, and attribute descriptions are maintained separately within the **Source Data Dictionary**, which serves as the authoritative reference for individual table structures.

---

# 5.2 Table Classification

The operational database tables are grouped according to their primary business function.

| Category        | Tables                                                       |
| --------------- | ------------------------------------------------------------ |
| User Management | `users`, `user_subscriptions`                                |
| Music Catalog   | `artists`, `albums`, `songs`, `playlists`                    |
| Podcast Catalog | `podcasts`, `podcast_episodes`                               |
| User Engagement | `sessions`, `listening_history`, `podcast_listening_history` |
| Commerce        | `payments`                                                   |
| Marketing       | `marketing_campaigns`                                        |

This classification reflects the functional organization of the Pulse Music application rather than the implementation order of the database.

---

# 5.3 Table Inventory

The operational database consists of the following business tables.

| Table                       | Primary Business Purpose                                |
| --------------------------- | ------------------------------------------------------- |
| `users`                     | Stores registered user accounts and profile information |
| `artists`                   | Stores artist information                               |
| `albums`                    | Stores music album metadata                             |
| `songs`                     | Stores individual song metadata                         |
| `playlists`                 | Stores user-created and editorial playlists             |
| `podcasts`                  | Stores podcast series information                       |
| `podcast_episodes`          | Stores podcast episode metadata                         |
| `sessions`                  | Records user application sessions                       |
| `listening_history`         | Records song playback events                            |
| `podcast_listening_history` | Records podcast playback events                         |
| `user_subscriptions`        | Records subscription lifecycle events                   |
| `payments`                  | Stores subscription payment transactions                |
| `marketing_campaigns`       | Stores marketing campaign performance data              |

Each table represents either a core business entity or a transactional business event.

---

# 5.4 Table Types

The operational tables can also be classified according to their data characteristics.

| Table Type         | Description                                        | Examples                                                                                       |
| ------------------ | -------------------------------------------------- | ---------------------------------------------------------------------------------------------- |
| Master Data        | Relatively stable business entities                | `users`, `artists`, `albums`, `songs`, `podcasts`                                              |
| Reference Data     | Lookup-style business information                  | *(No dedicated reference tables in this implementation)*                                       |
| Transaction Data   | Business events generated by user activity         | `payments`, `sessions`, `listening_history`, `podcast_listening_history`, `user_subscriptions` |
| Configuration Data | Operational configuration and campaign definitions | `marketing_campaigns`                                                                          |

Understanding these classifications helps determine update frequency, indexing strategy, and downstream ETL processing.

---

Each operational table is considered the authoritative source for its corresponding business entity.

| Table Category  | Data Owner           |
| --------------- | -------------------- |
| User Management | User Services        |

---

# 5.7 Data Retention

The operational database retains historical transactional records to support downstream analytics.

General retention principles include:

* User activity is retained for historical analysis.
* Listening events are not deleted after processing.
* Payment transactions remain permanently available.
* Subscription lifecycle events preserve historical changes.
* Music and podcast metadata remain available even after content becomes inactive.

For the purposes of this project, no automated archival or purge policy is implemented.

# 5.9 Implementation Order

To satisfy foreign key dependencies, tables should be created in the following order.

1. `artists`
2. `albums`
3. `songs`
4. `users`
5. `playlists`
6. `podcasts`
7. `podcast_episodes`
8. `marketing_campaigns`
9. `user_subscriptions`
10. `payments`
11. `sessions`
12. `listening_history`
13. `podcast_listening_history`

This sequence minimizes dependency conflicts during database deployment.

---

# 5.10 Table Specifications Summary

The operational database comprises thirteen normalized tables that collectively model the core business operations of the Pulse Music platform.

These tables provide the transactional foundation for the analytics pipeline, while detailed structural definitions are maintained within the Source Data Dictionary to ensure consistency, maintainability, and clear separation of documentation responsibilities.
