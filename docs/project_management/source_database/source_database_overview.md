# Source Database Specification (SDS)

## 1. Overview

### 1.1 Purpose

The Source Database Specification (SDS) defines the design and implementation of the operational database for the Pulse Music Analytics Platform.

The operational database simulates the transactional system used by a fictional music streaming service. It serves as the origin of all data processed by the analytics platform and provides the source data for the ETL pipeline.

This document describes the database architecture, schema design, table structures, constraints, indexing strategy, and data generation approach used to create a realistic operational environment for analytical processing.

---

## 1.2 Objectives

The primary objectives of the source database are to:

* Simulate a production-grade transactional database for a music streaming platform.
* Provide realistic source data for ETL development and testing.
* Support downstream Bronze, Silver, and Gold layer processing.
* Demonstrate best practices in relational database design.
* Serve as the operational source system for business intelligence and analytics.

---

## 1.3 Scope

The Source Database includes operational tables representing the core business processes of the Pulse Music platform.

These processes include:

* User management
* Music catalog management
* Podcast management
* Playlist management
* Music listening activity
* Podcast listening activity
* User sessions
* Subscription lifecycle events
* Payment transactions
* Marketing campaign management

The database is designed exclusively as a transactional source system and is not intended to support analytical reporting directly.

## 1.6 Operational Database Overview

The operational database represents the day-to-day activities of a fictional audio streaming service.

It captures transactional events generated by users interacting with the platform, including:

* Creating user accounts
* Streaming songs
* Listening to podcast episodes
* Managing playlists
* Purchasing subscriptions
* Processing payments
* Recording application sessions
* Tracking marketing campaign performance

The operational database is optimized for transaction processing (OLTP), while analytical workloads are performed exclusively within the downstream data warehouse.

---

## 1.7 Design Principles

The operational database has been designed according to the following principles:

### Normalized Structure

Operational tables are normalized to reduce redundancy and maintain data integrity.

### Referential Integrity

Primary and foreign key relationships ensure consistency across business entities.

### Transactional Focus

The schema is optimized for high-volume transactional operations rather than analytical queries.

### Realistic Data Simulation

Synthetic data is generated to closely resemble the scale, distribution, and quality characteristics of a real-world music streaming platform.

Intentional data quality issues—such as missing values, duplicate records, inconsistent formatting, and invalid values—are introduced in controlled scenarios to support ETL cleansing and validation exercises.

### Extensibility

The schema is designed to support future enhancements, including additional content types, user engagement features, and business processes.
