# 6. Constraints and Data Integrity

## 6.1 Purpose

This section defines the mechanisms used to preserve data integrity within the Pulse Music operational database.

Database constraints are used to prevent structurally invalid data from entering the source system while allowing realistic operational data-quality issues to exist where appropriate.

Validation responsibilities are divided between:

* Database-level constraints
* Application-level validation
* ETL-level data quality validation

This layered approach preserves operational integrity while supporting realistic downstream data cleansing and transformation.

---

## 6.2 Data Integrity Strategy

The operational database applies constraints according to the following principle:

> The database enforces structural and relational integrity, while complex business and analytical validation is performed downstream.

The database is responsible for preventing conditions such as:

* Duplicate primary keys
* Invalid foreign key references
* Missing mandatory values
* Structurally invalid numeric values
* Impossible date relationships that can be evaluated within a single row

The ETL pipeline is responsible for identifying conditions such as:

* Duplicate business entities
* Cross-table inconsistencies
* Suspicious but technically valid values
* Inconsistent categorical values
* Historical lifecycle conflicts
* Analytical anomalies

---

## 6.3 Primary Key Constraints

Every operational table contains a primary key that uniquely identifies each record.

| Table                       | Primary Key           |
| --------------------------- | --------------------- |
| `users`                     | `user_id`             |
| `artists`                   | `artist_id`           |
| `albums`                    | `album_id`            |
| `songs`                     | `song_id`             |
| `playlists`                 | `playlist_id`         |
| `podcasts`                  | `podcast_id`          |
| `podcast_episodes`          | `episode_id`          |
| `sessions`                  | `session_id`          |
| `listening_history`         | `playback_id`         |
| `podcast_listening_history` | `podcast_playback_id` |
| `user_subscriptions`        | `subscription_id`     |
| `payments`                  | `payment_id`          |
| `marketing_campaigns`       | `campaign_id`         |

All primary keys use UUID values generated by the operational system.

Primary key columns are:

* Unique
* Non-nullable
* Immutable after creation

## 6.4 Foreign Key Constraints

Foreign keys enforce relationships between operational entities.

| Child Table                 | Foreign Key       | Parent Table         | Referenced Key    |
| --------------------------- | ----------------- | -------------------- | ----------------- |
| `albums`                    | `artist_id`       | `artists`            | `artist_id`       |
| `songs`                     | `artist_id`       | `artists`            | `artist_id`       |
| `songs`                     | `album_id`        | `albums`             | `album_id`        |
| `playlists`                 | `user_id`         | `users`              | `user_id`         |
| `podcast_episodes`          | `podcast_id`      | `podcasts`           | `podcast_id`      |
| `sessions`                  | `user_id`         | `users`              | `user_id`         |
| `listening_history`         | `user_id`         | `users`              | `user_id`         |
| `listening_history`         | `song_id`         | `songs`              | `song_id`         |
| `listening_history`         | `playlist_id`     | `playlists`          | `playlist_id`     |
| `podcast_listening_history` | `user_id`         | `users`              | `user_id`         |
| `podcast_listening_history` | `episode_id`      | `podcast_episodes`   | `episode_id`      |
| `user_subscriptions`        | `user_id`         | `users`              | `user_id`         |
| `payments`                  | `subscription_id` | `user_subscriptions` | `subscription_id` |
| `payments`                  | `user_id`         | `users`              | `user_id`         |

Nullable foreign keys are permitted where the underlying business relationship is optional.

Examples include:

* `songs.album_id` for standalone singles
* `playlists.user_id` for platform-managed playlists
* `listening_history.playlist_id` for playback events originating outside playlists

---

## 6.5 Referential Actions

The operational database follows a conservative referential action strategy.

### ON DELETE

Foreign keys use:

```text
ON DELETE RESTRICT
```

or the PostgreSQL default:

```text
NO ACTION
```

for most relationships.

This prevents parent records from being deleted when dependent transactional or catalog records exist.

Historical business records should generally be deactivated rather than physically deleted.

### ON UPDATE

Primary business identifiers are treated as immutable.

Foreign keys therefore use:

```text
ON UPDATE NO ACTION
```

Primary key values are not expected to change after record creation.

---

## 6.6 Unique Constraints

Unique constraints are applied only where business uniqueness can be reliably enforced.

| Table   | Column  | Constraint |
| ------- | ------- | ---------- |
| `users` | `email` | Unique     |

Other descriptive attributes are not required to be unique.

Examples include:

* Artist names
* Album titles
* Song titles
* Podcast titles
* Playlist names
* Campaign names

Multiple legitimate records may share these values.

---

## 6.7 NOT NULL Constraints

`NOT NULL` constraints are applied to attributes required for the operational record to be meaningful.

Typical mandatory fields include:

* Primary keys
* Required foreign keys
* Core business identifiers
* Transaction timestamps
* Required status fields
* Audit timestamps

Optional descriptive attributes remain nullable where missing information is operationally acceptable.

Examples include:

* User city
* Artist country
* Song genre
* Playlist description
* Podcast category
* Session end timestamp
* Payment method

Nullability requirements for individual columns are defined in the Source Data Dictionary.

---

## 6.8 Check Constraints

Check constraints enforce simple row-level validation rules.

Examples include:

| Table                       | Validation Rule                                |
| --------------------------- | ---------------------------------------------- |
| `songs`                     | `duration_seconds > 0`                         |
| `podcast_episodes`          | `duration_seconds > 0`                         |
| `podcast_episodes`          | `episode_number > 0` when populated            |
| `sessions`                  | End timestamp must not precede start timestamp |
| `listening_history`         | `seconds_played >= 0`                          |
| `podcast_listening_history` | `seconds_played >= 0`                          |
| `user_subscriptions`        | End date must not precede start date           |
| `payments`                  | `amount >= 0`                                  |
| `marketing_campaigns`       | Budget and spend must be non-negative          |
| `marketing_campaigns`       | Performance counts must be non-negative        |

Check constraints are limited to deterministic rules that can be evaluated using values within the same row.

---

## 6.9 Database-Level Validation

The operational database directly enforces rules required to preserve structural integrity.

Examples include:

* Primary key uniqueness
* Required field presence
* Valid foreign key references
* Non-negative transaction amounts
* Positive content durations
* Valid chronological relationships within a single record

Records violating these rules are rejected by PostgreSQL.

---

## 6.10 ETL-Level Validation

Some business rules require comparison across records or tables and are therefore validated within the ETL pipeline.

Examples include:

### Playback Validation

* `seconds_played` exceeds the associated song duration
* Podcast listening duration exceeds episode duration
* Contradictory completion and skip indicators

### Subscription Validation

* Overlapping subscription periods
* Invalid subscription lifecycle sequences
* Status and auto-renew inconsistencies

### Payment Validation

* Payment user does not match subscription owner
* Payment occurs before subscription start date
* Unexpected payment amounts for the associated plan

### Marketing Validation

* Clicks exceed impressions
* Conversions exceed clicks
* Campaign spend significantly exceeds budget

### Entity Validation

* Duplicate business entities
* Inconsistent categorical values
* Invalid country names
* Unexpected future dates

These records may be technically valid from a database perspective while still requiring cleansing, rejection, or investigation during ETL processing.

---

## 6.11 Intentional Data Quality Anomalies

The synthetic source data includes controlled anomalies to simulate realistic data engineering conditions.

However, intentional anomalies must not require the removal of essential database integrity controls.

Therefore, anomalies are introduced using two approaches.

### Operationally Valid Anomalies

These can exist directly within the source database.

Examples include:

* Missing optional values
* Inconsistent capitalization
* Leading and trailing whitespace
* Duplicate business entities with different primary keys
* Invalid categorical labels
* Suspicious but structurally valid dates
* Cross-record inconsistencies

### Raw Extraction Anomalies

Anomalies that would violate enforced database constraints may be introduced after extraction into the Bronze layer or through controlled raw-source files.

Examples include:

* Orphaned foreign keys
* Negative values blocked by check constraints
* Duplicate primary keys
* Invalid same-row chronological relationships

This approach preserves the credibility of the operational database while still allowing the ETL pipeline to demonstrate robust data-quality handling.

## 6.12 Constraint Naming Standards

Constraints follow the naming conventions established in Section 3.

| Constraint Type | Naming Pattern                  |
| --------------- | ------------------------------- |
| Primary Key     | `pk_<table>`                    |
| Foreign Key     | `fk_<table>_<referenced_table>` |
| Unique          | `uq_<table>_<column>`           |
| Check           | `chk_<table>_<rule>`            |

Examples include:

```text
pk_users

fk_albums_artists

uq_users_email

chk_songs_duration_positive

chk_sessions_timestamp_order
```

Where multiple foreign keys reference the same table or additional clarification is required, the foreign key column may be included in the constraint name.

---

## 6.13 Constraint Implementation Principles

The following principles govern constraint implementation:

1. Constraints must protect essential structural integrity.
2. Constraints should not attempt to encode every possible business rule.
3. Complex cross-table validation belongs in application or ETL logic.
4. Historical transactional records should be preserved.
5. Physical deletion of referenced business entities should be restricted.
6. Intentional data-quality testing must not unnecessarily weaken the operational schema.
7. Constraint definitions must remain consistent with the Source Data Dictionary and ERD.

---

## 6.14 Constraints and Data Integrity Summary

The Pulse Music operational database uses primary keys, foreign keys, unique constraints, nullability rules, and check constraints to maintain reliable structural and relational integrity.

Simple deterministic rules are enforced directly by PostgreSQL, while complex business validation and analytical data-quality checks are handled by the downstream ETL pipeline.

This layered approach provides a realistic balance between source-system integrity and the imperfect data conditions commonly encountered in production analytics environments.
