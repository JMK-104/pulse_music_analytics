# 4. Source Data Model

## 4.1 Purpose

This section describes the logical organization of the Pulse Music operational database and the business entities represented within it.

The source data model is designed using a normalized relational structure to support efficient Online Transaction Processing (OLTP). Each table represents a discrete business entity or operational event generated by the Pulse Music platform.

The source data model serves as the foundation for the downstream ETL pipeline and analytics warehouse.

---

# 4.2 Source Data Model Overview

The operational database consists of thirteen core business tables grouped into five functional domains.

| Business Domain      | Tables                                                       |
| -------------------- | ------------------------------------------------------------ |
| User Management      | `users`, `user_subscriptions`                                |
| Music Catalog        | `artists`, `albums`, `songs`, `playlists`                    |
| Podcast Catalog      | `podcasts`, `podcast_episodes`                               |
| User Engagement      | `sessions`, `listening_history`, `podcast_listening_history` |
| Commerce & Marketing | `payments`, `marketing_campaigns`                            |

Each domain represents a major operational area of the Pulse Music platform while remaining fully integrated through primary and foreign key relationships.

---

# 4.3 Business Entity Summary

The following tables comprise the operational source database.

| Table                       | Description                                  |
| --------------------------- | -------------------------------------------- |
| `users`                     | Registered users of the Pulse Music platform |
| `artists`                   | Music artists and performers                 |
| `albums`                    | Music albums released by artists             |
| `songs`                     | Individual songs available for streaming     |
| `playlists`                 | User-created and editorial playlists         |
| `podcasts`                  | Podcast series available on the platform     |
| `podcast_episodes`          | Individual podcast episodes                  |
| `sessions`                  | User application sessions                    |
| `listening_history`         | Song playback events                         |
| `podcast_listening_history` | Podcast playback events                      |
| `user_subscriptions`        | Subscription lifecycle events                |
| `payments`                  | Subscription payment transactions            |
| `marketing_campaigns`       | Marketing campaign performance records       |

---

# 4.4 Entity Dependency Hierarchy

The source tables are implemented according to their dependency relationships to preserve referential integrity.

```text
artists
    │
    ▼
albums
    │
    ▼
songs

podcasts
    │
    ▼
podcast_episodes

users
    │
    ├──────────────┐
    ▼              ▼
playlists      user_subscriptions
    │              │
    │              ▼
    │          payments
    │
    ├──────────────┐
    ▼              ▼
sessions    listening_history
                   │
                   ▼
podcast_listening_history

marketing_campaigns
```

Independent entities are created before dependent entities during database implementation.

# 4.5 Entity Relationships

The principal business relationships within the operational database are summarized below.

| Parent Entity      | Child Entity              | Relationship |
| ------------------ | ------------------------- | ------------ |
| Artists            | Albums                    | One-to-Many  |
| Albums             | Songs                     | One-to-Many  |
| Artists            | Songs                     | One-to-Many  |
| Users              | Playlists                 | One-to-Many  |
| Users              | Sessions                  | One-to-Many  |
| Users              | Listening History         | One-to-Many  |
| Users              | User Subscriptions        | One-to-Many  |
| User Subscriptions | Payments                  | One-to-Many  |
| Podcasts           | Podcast Episodes          | One-to-Many  |
| Podcast Episodes   | Podcast Listening History | One-to-Many  |

These relationships ensure consistent representation of business entities while supporting downstream ETL transformations.

---

# 4.6 Source Data Characteristics

The operational database captures transactional business events generated by normal platform activity.

Typical characteristics include:

| Characteristic          | Description                                                   |
| ----------------------- | ------------------------------------------------------------- |
| High transaction volume | Continuous user interactions generate new records.            |
| Normalized structure    | Data redundancy is minimized.                                 |
| Frequent inserts        | Listening events and sessions are continuously created.       |
| Periodic updates        | User profiles and subscriptions may change over time.         |
| Referential integrity   | Parent-child relationships are enforced through foreign keys. |

The database is optimized for transactional consistency rather than analytical performance.

# 4.8 Source Data Quality Characteristics

The source database intentionally contains controlled data quality issues to simulate real-world operational environments.

Examples include:

* Duplicate records
* Missing optional values
* Inconsistent text capitalization
* Leading and trailing whitespace
* Invalid categorical values
* Orphaned foreign key references (where appropriate for ETL testing)
* Future-dated timestamps
* Negative numeric values in selected test records
* Incorrect country names and abbreviations

These anomalies support the implementation and demonstration of data cleansing, validation, and exception handling within the ETL pipeline.

---

# 4.9 Relationship to the Analytics Warehouse

The operational database is not used directly for reporting.

Instead, it serves as the source system for the Medallion Architecture.

```text
Operational Database
        │
        ▼
Bronze Layer
        │
        ▼
Silver Layer
        │
        ▼
Gold Layer
        │
        ▼
Galaxy Schema
        │
        ▼
Business Intelligence & Analytics
```

All analytical queries, dashboards, and KPI calculations are performed against the Gold Layer rather than the operational database.

---

# 4.10 Source Data Model Summary

The Pulse Music operational database consists of thirteen normalized business tables organized into functional domains that reflect the core activities of a modern audio streaming platform.

This data model provides a realistic transactional environment from which the ETL pipeline extracts, cleanses, and transforms operational data into the enterprise analytics warehouse, ensuring a robust foundation for business intelligence and advanced analytics.
