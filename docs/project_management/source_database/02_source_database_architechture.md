# 2. System Architecture

## 2.1 Architecture Overview

The Pulse Music Analytics Platform follows a layered data architecture that separates operational transaction processing from analytical reporting.

The operational database serves as the authoritative source system, capturing all day-to-day business transactions generated by the Pulse Music application.

Data from the operational database is extracted through the ETL pipeline and progressively transformed into the analytics warehouse using a Medallion Architecture consisting of Bronze, Silver, and Gold layers.

This separation ensures that transactional workloads do not interfere with analytical processing while providing a scalable foundation for business intelligence and reporting.

---

## 2.2 High-Level System Architecture

The operational database occupies the first layer of the analytics platform.

```text
                        Pulse Music Application
                                  │
                                  ▼
                    Operational PostgreSQL Database
                                  │
                                  ▼
                           Bronze Layer (Raw)
                                  │
                                  ▼
                      Silver Layer (Clean & Standardized)
                                  │
                                  ▼
                      Gold Layer (Galaxy Schema Warehouse)
                                  │
                                  ▼
                  SQL Analytics • Tableau Dashboards • KPIs
```

The operational database is optimized for Online Transaction Processing (OLTP), while all analytical workloads are performed exclusively against the Gold Layer.

---

## 2.3 Operational Database Role

The operational database is responsible for recording business transactions generated by the Pulse Music platform.

Core responsibilities include:

* Managing user accounts
* Maintaining the music catalog
* Maintaining the podcast catalog
* Recording listening activity
* Tracking user sessions
* Managing playlists
* Recording subscription events
* Processing payment transactions
* Managing marketing campaign information

The operational database is considered the system of record for all transactional data.

---

## 2.4 Architectural Characteristics

The source database has been designed according to the following architectural principles.

| Characteristic      | Description                                   |
| ------------------- | --------------------------------------------- |
| Database Type       | Relational Database Management System (RDBMS) |
| Processing Model    | Online Transaction Processing (OLTP)          |
| Data Model          | Normalized Relational Model                   |
| Primary Purpose     | Capture operational business transactions     |
| Data Ownership      | Source System of Record                       |
| Reporting Usage     | Not intended for analytical reporting         |
| Downstream Consumer | ETL Pipeline                                  |

---

## 2.5 Database Technology

The operational database is implemented using PostgreSQL.

| Component             | Technology                                                     |
| --------------------- | -------------------------------------------------------------- |
| Database Engine       | PostgreSQL                                                     |
| SQL Standard          | ANSI SQL (PostgreSQL-compatible)                               |
| Character Encoding    | UTF-8                                                          |
| Date/Time Standard    | UTC                                                            |
| Primary Key Strategy  | Natural business identifiers (UUIDs)                           |
| Referential Integrity | Foreign Key Constraints                                        |
| Indexing              | B-tree indexes on primary and frequently queried business keys |

The selected technology provides robust transactional support, strong ACID compliance, and compatibility with modern analytics engineering workflows.

---

## 2.6 Operational Data Flow

Business transactions originate within the Pulse Music application and are persisted in the operational database.

The ETL pipeline periodically extracts transactional data according to the incremental loading strategy defined in the ETL Mapping Specification.

The data flow follows the sequence below.

```text
User Activity
      │
      ▼
Pulse Music Application
      │
      ▼
Operational Database
      │
      ▼
ETL Extraction
      │
      ▼
Bronze Layer
      │
      ▼
Silver Layer
      │
      ▼
Gold Warehouse
      │
      ▼
Business Intelligence
```

This architecture ensures a clear separation between operational processing and analytical workloads.

---

## 2.7 Source System Boundaries

The operational database is intentionally limited to transactional processing.

The following activities are outside the scope of the source system:

* Aggregated reporting
* Business intelligence dashboards
* Historical dimensional modeling
* Advanced analytical calculations
* KPI computation
* Data warehouse transformations

These responsibilities are performed within the downstream analytics warehouse.

---

## 2.8 Design Assumptions

The architecture is based on the following assumptions.

* The operational database represents a single production environment.
* Source data is considered the authoritative version of each business event.
* Operational tables are normalized to reduce redundancy.
* Analytical transformations occur only after data enters the ETL pipeline.
* Source tables may contain realistic data quality issues to support ETL validation exercises.
* All timestamps are stored in Coordinated Universal Time (UTC).

These assumptions guide both the database implementation and the downstream warehouse design.

---

## 2.9 System Architecture Summary

The operational PostgreSQL database provides the transactional foundation of the Pulse Music Analytics Platform.

By separating operational processing from analytical workloads and serving as the authoritative source for the ETL pipeline, the source database enables the construction of a scalable, maintainable, and production-oriented analytics warehouse while preserving the integrity of day-to-day business operations.
