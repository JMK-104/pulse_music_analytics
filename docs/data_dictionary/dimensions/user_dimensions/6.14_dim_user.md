# 6.14 Dimension Table — `dim_user`

## Business Purpose

The `dim_user` table stores descriptive information about Pulse Music users.

It is one of the primary conformed dimensions in the warehouse and supports analysis across listening behavior, subscriptions, payments, sessions, and podcast engagement.

This dimension uses Slowly Changing Dimension Type 2 (SCD Type 2) methodology to preserve historical user attributes.

---

## Grain

**One row represents one version of a user profile.**

Example:

| user_key | user_id | country       | is_current |
| -------- | ------- | ------------- | ---------- |
| 101      | U123    | Canada        | FALSE      |
| 205      | U123    | United States | TRUE       |

---

## Source

Derived from the operational `Users` table during ETL.

`user_key`

## Column Definitions

| Column               | Data Type    | Nullable |  PK |  FK | Default        | Description                                  |
| -------------------- | ------------ | :------: | :-: | :-: | -------------- | -------------------------------------------- |
| user_key             | INTEGER      |    No    |  ✓  |     | Auto-generated | Surrogate key.                               |
| user_id              | VARCHAR(20)  |    No    |     |     |                | Business identifier from operational system. |
| signup_date          | DATE         |    No    |     |     |                | Date user registered.                        |
| signup_channel       | VARCHAR(50)  |    No    |     |     |                | User acquisition channel.                    |

| effective_start_date | DATE         |    No    |     |     |                | Date this record became active.              |
| effective_end_date   | DATE         |    Yes   |     |     | NULL           | Date this record stopped being active.       |
| is_current           | BOOLEAN      |    No    |     |     | TRUE           | Indicates current user record.               |

---

## Business Rules

* Each user can have multiple historical records.
* Only one record per user may have `is_current = TRUE`.
* Changes to tracked attributes create a new version.
* Historical fact records must join to the correct user version.

Tracked attributes:

* Country
* City
* Age group
* Gender
