# 7.4 Fact Table — `fact_payment`

## Business Purpose

The `fact_payment` table records all subscription-related payment transactions processed by the Pulse Music platform.

It enables financial reporting, recurring revenue analysis, payment success rates, refund analysis, and subscription revenue tracking across plans and payment methods.

---

## Grain

**One row represents one payment transaction.**

Example:

A user renews their Premium subscription.

↓

One payment is processed.

↓

One record is created.

---

## Source

Derived from the operational `Payments` table during ETL.

---

## Primary Key

`payment_key`

---

## Foreign Keys

| Foreign Key        | References         |
| ------------------ | ------------------ |
| date_key           | dim_date           |
| user_key           | dim_user           |
| plan_key           | dim_plan           |
| payment_method_key | dim_payment_method |

---

## Measures

| Measure         | Description                                               |
| --------------- | --------------------------------------------------------- |
| payment_amount  | Total amount charged for the transaction.                 |
| successful_flag | Indicates whether the payment was successfully processed. |
| refunded_flag   | Indicates whether the payment was refunded.               |

---

## Column Definitions

| Column             | Data Type     | Nullable |  PK |  FK | Default        | Description                                                  |
| ------------------ | ------------- | :------: | :-: | :-: | -------------- | ------------------------------------------------------------ |
| payment_key        | BIGINT        |    No    |  ✓  |     | Auto-generated | Surrogate key uniquely identifying each payment transaction. |
| date_key           | INTEGER       |    No    |     |  ✓  |                | References `dim_date`.                                       |
| user_key           | INTEGER       |    No    |     |  ✓  |                | References `dim_user`.                                       |
| plan_key           | INTEGER       |    No    |     |  ✓  |                | References `dim_plan`.                                       |
| payment_method_key | INTEGER       |    No    |     |  ✓  |                | References `dim_payment_method`.                             |
| payment_amount     | NUMERIC(10,2) |    No    |     |     |                | Amount charged to the customer.                              |
| successful_flag    | BOOLEAN       |    No    |     |     | TRUE           | TRUE if the payment was successfully processed.              |
| refunded_flag      | BOOLEAN       |    No    |     |     | FALSE          | TRUE if the payment was refunded.                            |

---

## Business Rules

* One row represents one payment transaction.
* Every payment must reference one user, one subscription plan, and one payment method.
* Payment amount must be greater than zero.
* A refunded payment must have originally been successful.
* Failed payments cannot be marked as refunded.

---

## Data Quality Rules

| Rule                                                   | Expected Result |
| ------------------------------------------------------ | --------------- |
| No duplicate payment_key values                        | Pass            |
| Payment amount > 0                                     | Pass            |
| All foreign keys exist                                 | Pass            |
| Refunded payments are successful                       | Pass            |
| Successful and refunded flags are logically consistent | Pass            |

---

## Estimated Size

Approximately **300,000–500,000 rows**.
